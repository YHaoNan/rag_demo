# 背景与目标
+ **背景**：期望以自动化的代码审查方案，帮助发现代码仓库中潜在的问题项，避免其造成严重损失。
+ **目标**：调研常见的基于规则的**静态代码审查（SAST）**方案以及基于**大语言模型的智能代码审查**方案在能力差异、成本、常见集成形态方面的差异。



会议目标：

1. 说明调研结果，目前构想的方案
2. 拉齐目标
3. 确定方向

# 核心需求
:::warning
本节提出对于代码审查系统的需求，后续的技术调研将围绕实现这些需求展开。

:::

+ **全量扫描**：可以对历史存在的、庞大的代码库进行全量的扫描，发现历史已有问题。
+ **增量扫描**：基于PR、提交等，对增量的代码进行扫描，追求快速且实时的反馈。
+ **测试左移**：尽量将问题发现环节移动到前置过程中，如编码时、提交时，可以更提早的抑制问题。
+ **流水线集成**：可以集成至CI/CD流水线，并在发现严重问题时对代码合环节进行阻塞。
+ **统一系统**：静态方案和大模型方案能否整合至一套系统，统一进行问题管理。
+ **管理视图**：可否可以利用问题数据，对研发培训、质量管理提出参考性建议。

# 能力对比及调研
## 静态代码审查 vs. 大语言模型审查
:::warning
**TL; DR**

静态审查成本低，成熟，但难以发现具有上下文语义关联的问题；大语言模型方案则完全相反，二者在能力上呈现互补状态。

:::

| **方案** | **优势** | **劣势** |
| --- | --- | --- |
| 传统静态审查 | + 基于规则，成本低，速度快<br/>+ 相当成熟的流水线集成方案<br/>+ 久经验证的落地实践<br/>+ 完善的测试左移集成<br/>+ 精确，可预测 | + 无语义理解，难以发现业务相关问题<br/>    - 如当前流程不能开票，但由于状态机校验遗漏导致可能开票<br/>+ 对问题项缺乏理解能力，无法做知识沉淀来指导研发提效<br/>+ 无法识别上下文相关漏洞<br/>    - 如WorkService某接口以前约定只返回作业中的作业，其它下游Service利用了这个约定。在某次修改时改成了返回作业中和待措施落实的作业，导致下游直接炸裂<br/>    - A call B，B以前返回可变List，某次变更某个代码路径可能返回不可变List，下游直接瘫痪<br/>+ 定制规则会使用DSL |
| 大语言模型审查 | + 具有<u>业务语义理解</u>能力<br/>+ 可以识别上下文相关漏洞<br/>+ 擅长做知识总结沉淀<br/>+ 可基于自然语言定制规则 | + 基于算力，<u>成本高昂</u>，速度低下，几乎难以做全量扫描<br/>+ 当前没有特别成熟的方案，落地实践不够，这意味着可能需要更多的开发工作量<br/>+ 流水线集成约等于0<br/>+ 几乎和传统SAST都是割裂的两套系统<br/>+ <u>不可预测，幻觉问题难以解决</u>，无法直接作为拦截流水线的指导<br/>+ <u>代码数据出域</u>。若利用在线大模型，或直接使用在线评审服务，则难以避免代码数据出域问题<br/>+ 少量方案具有测试左移集成 |

## 大模型方案调研
:::warning
**TL; DR**

目前市面上成熟的大模型代码审查方案较少，基本上都提供PR级扫描、问题管理、PR评论总结等能力。

目前除了商用的云提供商方案，鲜有能与SAST系统整合的，基本都是独立系统。

:::

大模型代码审查，目前产品形态各异，我们调研了市面上比较常见的一些方案。目前比较成熟的有开源方案`Kodus AI`以及云提供商方案`Snyk Code`



| **方案** | **特点/优势** | **劣势** |
| --- | --- | --- |
| ✅Snyk Code | + 云方案<br/>+ SAST + 智能扫描一体，可集成至IDE、CLI、Git、CI/CD管道<br/>+ 完善的测试左移支持（各种插件），毫秒级响应 | + 成本高昂 $25/人/月 1000次测试<br/>+ 代码出域 |
| ✅Kodus AI | + 支持各种模型：Claude、GPT、GLM、Llama、OpenAI兼容（代码可以不出域）<br/>+ 开源以及商业版支持<br/>+ 完善的代码关联上下文发现支持<br/>+ 支持repo级自定义评审规则<br/>+ 开箱即用的Github、Gitlab、Bitbucket等PR集成<br/>+ 提供cli可以运行在CI/CD中，但支持不完善 | + 中文遵守比较差，提示词注入后还是经常出现英文<br/>+ 仅PR评论<br/>+ 没有测试左移支持 |
| ❌PR-Agent(Qodo) | + 完善的IDE插件、CICD、git支持<br/>+ 基于PR的审查 | + 仅基于PR的审查<br/>+ 比较简单，无跨项目，跨其它文件的审查能力 |
| ❌villesau/ai-codereviewer | + 可集成至Github Actions<br/>+ 基于PR的Code Review | + OpenAI模型绑定，无替换模型能力<br/>+ 仅支持PR<br/>+ 仅支持github actions<br/>+ 仅给出建议，需要人工处理<br/>+ 实现简单，维护缺失（3年未维护）<br/>+ 数据出域 |
| ❌Codex Security<br/> | + LLM原生架构<br/>+ 全系统理解力 -> 漏洞发现 -> 沙箱验证 | + 尚处于预览版<br/>+ 文档尚不完善，功能列表尚不清晰<br/>+ 没有CICD等流程集成 |
| ❌Claude Code Security | + LLM原生架构<br/>+ 内置在Claude Code中 | + 使用形态不符合要求，没有CICD等流程集成 |

## KodusAI二开 or 自建
自建周期长，KodusAI笨重且技术栈不一致

## 测试左移是否有必要
测试左移是一个领域内非常常见的话题以及最佳实践，问题发现的环节越晚，其改造成本、管理成本将越高。

常见的测试左移工具可以集成至IDE中，在编程时、代码单次提交时完成检查。

+ SonarQube有完善的测试左移支持
+ Kodus AI、自建平台都没有，可能需要自己开发

## 如何控制大模型审查的成本问题
+ **仅提供PR-based审查**（小型规模PR单次十万级~百万级token，成本0.5~2元左右）
+ **上下文复用**：代码中经常出现小型改动（如仅单行），但要携带大量通用上下文，此时应合并审查
+ **Light/Heavy双路审查**：默认首次开启轻量审查，不携带过多上下文，仅在模型返回需要更多上下文做判断时，采用重型审查

## 大型文件如何装进上下文
虽然顶级模型能达到兆级别的上下文窗口，但常见的模型仍只有128K，此窗口对于大部分代码评审需求足够，极端情况可以采取chunk方案

+ 总窗口 - 静态提示词预算 = 剩余上下文窗口
+ 若文件大小 > 剩余上下文窗口，使用AST基于完整代码语义进行chunk分片，分片后再多次调用模型
+ 若仍超出，放弃并记录错误

## 如何精确的在大模型审查过程中拉取相关上下文
可以基于AST技术对代码仓库进行代码地图生成，实测基于Tree-sitter构建的AST扫描器在扫描小型仓库时（自建代码评审平台）仅需秒级时间。
